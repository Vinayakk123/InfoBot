# Tech Stack

Every value below is read directly from the code (not assumed). File references point to the exact line where each choice is made.

## Frontend: Streamlit

**What's used:** `streamlit` chat UI in [streamlit_app.py](../streamlit_app.py) — `st.chat_input`/`st.chat_message` for the conversation, a sidebar for top-k control and a "clear conversation" button, `st.cache_resource` to load the RAG engine once per process, and `st.session_state` + a local `chat_history.json` file for persistence across restarts.

- **Why it was likely chosen:** Fastest way to ship a working chat UI in pure Python with no separate frontend build step — appropriate for an internal demo/single-user tool.
- **Alternatives and tradeoffs:** Gradio — similarly fast to build, slightly different widget model, comparable tradeoffs. A dedicated frontend (React/Next.js) talking to a FastAPI backend — much more setup and a real frontend build, but needed if this becomes a multi-user product with custom UX, auth, or embedding into another site. **Reach for FastAPI + a real frontend once you need proper multi-user auth, custom branding, or to embed the chat into an existing client portal**; stick with Streamlit for internal tools and quick client demos.
- **Known limitations in this codebase:** No authentication — anyone who can reach the running app can query it and consume the configured Groq API key's quota ([docs/02-ARCHITECTURE.md](02-ARCHITECTURE.md)). Single-process design means one Streamlit server handles all users; it's not built for concurrent multi-user load.

## LLM: Groq (`llama-3.1-8b-instant`)

**What's used:** `ChatGroq` from `langchain-groq`, instantiated with `model_name=llm_model` where `llm_model` defaults to `"llama-3.1-8b-instant"` ([src/search.py:13](../src/search.py#L13), also hardcoded as `LLM_MODEL` in [streamlit_app.py:27](../streamlit_app.py#L27)). The API key is read from `GROQ_API_KEY` via `os.getenv` ([src/search.py:25](../src/search.py#L25)).

- **Why it was likely chosen:** Groq's hosted inference is very low-latency and has a free/inexpensive tier, and `llama-3.1-8b-instant` is a small, fast model — a good fit for quick demo responses without running any LLM infrastructure yourself.
- **Alternatives and tradeoffs:** OpenAI (`gpt-4o-mini` or similar) — broader tooling/ecosystem support, but a different pricing model and no local/on-prem option. Anthropic Claude — stronger reasoning quality for complex questions, at higher latency/cost than an 8B model. A self-hosted model via Ollama — needed if the client has an on-prem/no-external-API-calls requirement, at the cost of needing your own GPU/CPU capacity and losing Groq's inference speed. **For a client with strict data-residency or no-external-API constraints, swap in Ollama or another on-prem-servable model; for a client needing higher-quality reasoning and willing to pay more, swap in a larger hosted model** — the swap is isolated to `RAGSearch.__init__` and the `llm_model` parameter.
- **Known failure modes in this codebase:** `self.llm.invoke([prompt])` in [src/search.py](../src/search.py) has no retry/backoff or timeout handling of its own — if the Groq API rate-limits, times out, or the key is invalid/revoked, the exception propagates up. `streamlit_app.py`'s `handle_user_query` catches this at the UI layer with a broad `try/except`, logs the full traceback, and shows the user a generic "Sorry, I ran into an error" message rather than crashing the app — but there's no automatic retry, so a transient rate-limit or timeout requires the user to resubmit the question. `llama-3.1-8b-instant` is a small model; on longer or more nuanced multi-document questions, answer quality will trail a larger model.
- **Two LLM calls per follow-up question.** When conversation history is present, `RAGSearch.search_and_summarize_with_sources()` ([src/search.py:134](../src/search.py#L134)) first calls `_condense_query()` ([src/search.py:52](../src/search.py#L52)) to rewrite the question into a standalone one for retrieval, then makes the usual answer-generation call — a latency and token-cost tradeoff (roughly double) accepted to keep pronoun/implicit-reference follow-ups ("What does it do?") retrieving the right chunks. Turn 1 (no history yet) still makes only one call. See [05-TROUBLESHOOTING.md](05-TROUBLESHOOTING.md#conversational-memory).

## Embedding model: `sentence-transformers` (`all-MiniLM-L6-v2`)

**What's used:** `SentenceTransformer("all-MiniLM-L6-v2")`, loaded in both `EmbeddingPipeline` ([src/embedding.py:15](../src/embedding.py#L15)) and `FaissVectorStore` ([src/vectorstore.py:20](../src/vectorstore.py#L20)) — the same model name must be used at index-build time and query time since embeddings from different models aren't comparable.

- **Why it was likely chosen:** `all-MiniLM-L6-v2` is small (~80MB), fast on CPU, and needs no GPU or external embedding API — document content never leaves the machine during embedding, and there's no per-call cost or network dependency for this step.
- **Alternatives and tradeoffs:** A larger sentence-transformers model (e.g. `all-mpnet-base-v2`) — better embedding quality, meaningfully slower on CPU. OpenAI/Cohere hosted embedding APIs — offload compute and often better quality, but reintroduce an external API dependency and per-call cost, and document text would leave the machine (a problem if data residency matters). **For a client with strict local/offline requirements, this local model is already the right choice; for a client prioritizing retrieval accuracy over cost/latency and willing to send data to a third party, a hosted embedding API is the natural upgrade.**
- **Known failure modes in this codebase:** First run downloads the model from Hugging Face Hub (tens of MB) before any embedding can happen — the Dockerfile/compose setup accounts for this by mounting a named volume (`hf-cache` → `/home/appuser/.cache/huggingface`, see [docker-compose.yml](../docker-compose.yml)) so the model isn't re-downloaded every time the container is recreated, but the very first run (no cache yet) will be slower and requires outbound internet access to Hugging Face. There is no GPU code path in this codebase — encoding always runs on CPU, which is fine at the current small document/query scale but will become a bottleneck if chunk volume or query throughput grows significantly.

## Chunking strategy: `RecursiveCharacterTextSplitter`

**What's used:** `langchain_text_splitters.RecursiveCharacterTextSplitter` with `chunk_size=1000`, `chunk_overlap=200`, `length_function=len`, `separators=["\n\n", "\n", " ", ""]` — all defaults on `EmbeddingPipeline.__init__` ([src/embedding.py:12](../src/embedding.py#L12)) and mirrored as defaults on `FaissVectorStore.__init__` ([src/vectorstore.py:14](../src/vectorstore.py#L14)).

- **Why it was likely chosen:** Recursive character splitting with a fallback separator list (paragraph → line → word → character) is a robust general-purpose default that works reasonably across mixed document types (PDF text, plain text, CSV rows, Word docs) without any per-format tuning. A 1000-character chunk with 200-character overlap is a common starting point that balances retrieval precision (small enough that a chunk is topically focused) against having enough context per chunk for the LLM to answer from.
- **Alternatives and tradeoffs:** Semantic chunking (split on embedding-similarity boundaries rather than character count) — better topical coherence per chunk, more compute cost and complexity to implement. Format-specific splitters (e.g. Markdown header splitting, or row-based chunking for CSV/Excel rather than character-based) — would preserve structure better for those formats specifically, at the cost of format-specific splitting logic instead of one uniform splitter. **If a client's documents are highly structured (tables, forms, code), a format-aware splitter is worth the added complexity; for general prose-heavy document sets like this one, the current uniform approach is a reasonable default.**
- **Known failure modes in this codebase:** None specific to chunk boundaries were observed (no test suite exercises edge cases like a single word longer than 1000 characters), but there's no automated test coverage of chunking correctness — see [02-ARCHITECTURE.md](02-ARCHITECTURE.md) limitations.

## Vector database: FAISS (`IndexFlatL2`)

**What's used:** `faiss.IndexFlatL2` built and queried in `FaissVectorStore` ([src/vectorstore.py](../src/vectorstore.py)), persisted to disk as two files under `faiss_store/`: `faiss.index` (the FAISS index itself, via `faiss.write_index`) and `metadata.pkl` (a pickled Python list of per-chunk metadata dicts — source file, page/row, chunk text — kept index-aligned with the FAISS vectors).

- **Why it was likely chosen:** `IndexFlatL2` is exact (brute-force) nearest-neighbor search — no approximation, no index-tuning parameters to get wrong, and simplest to reason about correctness for a small document set. `faiss-cpu` requires no GPU and no separate database server to run/maintain; the whole store is just two local files.
- **Alternatives and tradeoffs:** An approximate FAISS index (`IndexIVFFlat`, `IndexHNSWFlat`) — much faster at large scale, at the cost of approximate (not guaranteed exact) results and extra tuning parameters (`nlist`, `nprobe`, etc.). A managed vector database (Pinecone, Weaviate, Qdrant Cloud) — handles scaling, replication, and concurrent writes for you, but adds hosting cost and an external network dependency in place of a local, zero-cost store. A self-hosted Chroma/Qdrant instance (notably, `chromadb` is already present in this repo's `requirements-dev.txt` from earlier prototyping in `notebook/pdfloader.ipynb`, though it's not used in production) — adds a proper client-server API and metadata filtering, at the cost of running an extra service. **For a client needing on-prem with no third-party network calls, self-hosted Chroma/Qdrant or a bigger local FAISS index are the natural next step up; for a client at real scale (millions of chunks, high query concurrency) or wanting managed infra, Pinecone/Weaviate/Qdrant Cloud trade hosting cost and an external dependency for not having to operate the scaling yourself.**
- **Known failure modes in this codebase:** No locking or transactionality around writes — `build_from_documents()` calls `save()`, which unconditionally overwrites both `faiss.index` and `metadata.pkl` in full ([src/vectorstore.py:44-50](../src/vectorstore.py#L44)); if two processes (or two runs of `app.py`) do this concurrently, one write can clobber the other, and because the FAISS index and the pickled metadata list are two separate files written independently, a crash between the two writes can leave them out of sync (vector count in the index no longer lines up with the metadata list, causing incorrect or out-of-range source citations). If the `faiss_store/` persist directory is deleted while the app is running, the in-memory index keeps serving queries until the process restarts; on the next restart, `RAGSearch.__init__` checks for `faiss.index`/`metadata.pkl` and, if missing, attempts to rebuild — but hits the cold-start import bug documented in [02-ARCHITECTURE.md](02-ARCHITECTURE.md).

## Document ingestion: PDF, TXT, CSV, XLSX, DOCX, JSON

**What's used:** `load_all_documents()` in [src/data_loader.py](../src/data_loader.py) recursively globs `data/` for each extension and loads it with a dedicated `langchain_community` loader: `PyPDFLoader` (PDF), `TextLoader` (TXT), `CSVLoader` (CSV), `UnstructuredExcelLoader` (XLSX), `Docx2txtLoader` (DOCX), `JSONLoader` (JSON). Each format's loop is wrapped in its own `try/except Exception`, so one bad/corrupt file is logged and skipped without stopping ingestion of the rest.

- **Why it was likely chosen:** Covers the realistic spread of document formats a company actually has on hand (reports as PDF, data as CSV/XLSX, structured exports as JSON, plain notes as TXT, policies as DOCX), using LangChain's existing loader ecosystem rather than writing custom parsers for each format.
- **Alternatives and tradeoffs:** `Unstructured`'s own higher-level loaders or a dedicated document-parsing service (e.g. LlamaParse) — better structure/table extraction from complex PDFs, at the cost of an extra dependency (or external API) and more setup. Adding more formats (HTML, Markdown, PPTX) — straightforward to add following the same per-format-loader pattern already established here, if a client's documents include those. **This loader set is a reasonable default for typical office documents; swap in a dedicated parsing service specifically when PDFs have complex tables/layouts that `pypdf`'s text extraction handles poorly.**
- **Known failure modes in this codebase:** The per-format `try/except` means a failure is silent from the ingestion process's point of view (logged, not raised) — a file that fails to load simply won't appear in the index, with only a log line as evidence. More specifically: `UnstructuredExcelLoader` and `JSONLoader` depend on the `unstructured` and `jq` packages respectively, but **neither package appears in `requirements.txt`, `requirements-dev.txt`, `pyproject.toml`, or `uv.lock`** in this repo. If those packages aren't present in the actual installed environment, XLSX/JSON ingestion will raise an `ImportError` per file, get caught by the existing `try/except`, and be logged — meaning those formats could silently ingest zero documents even though the code path for them exists. Verify XLSX/JSON ingestion actually produces chunks (check `logs/infobot.log` for `Found N XLSX/JSON file(s)` followed by a non-zero loaded count) if you rely on those formats.
